# MARKETING PLAN — 社会デバッガー 普及プラン

> 収益化はしない。目的は「多くのユーザーに使われた実績」を、クラウドエンジニア就活と
> CS博士課程出願の資料にすること。よって指標は売上ではなく、ユーザー数・リテンション・
> 引用/言及・データの揃った物語である。

## 0. 大原則

1. **宣伝ではなくコンテンツ。** アプリの宣伝ツイートは拡散しない。アプリで作った面白い結果
   （崩壊のGIF、DIAGNOSIS カード、週次シナリオの生存率）そのものをコンテンツとして流す。
2. **大量の無差別リプは絶対にしない。** スパム判定によるアカウント制限リスクがあり、研究者は
   営業的コールドリプに強い拒否反応を持つ。個別・少数・価値提供が原則。
3. **政局に自分から接続しない。** 選挙・炎上の渦中に運営側から乗り込むのは、P4で遮断したい
   外部ノードを自ら呼び込む行為。構造だけ語れば、分かる人には分かる。
4. **数字なき実績は実績にならない。** すべての施策は track イベントとアナリティクスで効果測定
   できる状態で打つ。

## 1. チャネル別戦略

### X（メイン）
- **定期コンテンツ（週1）**: 週替わりシナリオの結果スレッド。「今週のシナリオ、生存率23%。
  クリアした人は◯◯を守っていた」— 集合統計＋構造の学びをセットで。動画/GIF付き。
- **単発コンテンツ**: プリセットの変奏（「報道の独立性を1つ下げただけでこうなる」）、
  巻き戻し体験の紹介（「1つだけ過去に戻せるなら、あなたは何を守る？」）、開発の裏側。
- **ハッシュタグ**: 固定タグを1つ決めて全投稿に付ける（例: #社会デバッガー）。検索での
  発見可能性と、ユーザー投稿の集約が目的。
- **プロフィール**: 固定ポストにWeb版URL＋30秒動画。「思想」より「体験」を前に。

### LINE（日本の拡散の本命）
- 結果カード＋一言の共有先として家族・地域のグループLINEを想定した文言設計。
  防災文脈（PAGE 2系）はXよりLINEで回る。共有テンプレートに「防災」の語を含む変種を用意。

### Zenn / Qiita（就活の本命）
- 記事1「単一HTML 33万文字のシミュレーターをCapacitorでアプリ化した話」
  → **ドラフト済み: `docs/articles/zenn-01-capacitor.md`**
- 記事2「S3+CloudFront+CDK+GitHub Actions(OIDC)で、コンテンツ配信をCI/CDに載せた話」
  （Dockerを使わない理由の節を含める — 技術選定を語れる証明として採用側に効く）
  → **ドラフト済み: `docs/articles/zenn-02-aws-cdk-oidc.md`**
- 記事3「開発者自身が2拠点生活の意思決定にこのモデルを使った話」（真正性の物語。
  アプリ内には載せない個人的文脈は、こちらで語る）
- 各記事末尾にアプリ導線。記事公開日はXの投稿と連動させる。

### 学会・研究コミュニティ
- 計算社会科学会・人工知能学会などの大会・研究会のポスター/LT/デモ枠に応募。
  「教育用ABMデモ」として出すのが通りやすい。採択自体が出願資料になる。
- 発表者・参加者との対面接点が、X経由より質の高い研究者ユーザーを生む。

### 教育チャネル
- 高校「公共」「情報I」・探究学習の教員コミュニティ。教員1人＝生徒30〜40人の乗数。
- 授業で使える1枚モノ（使い方＋問いの例）をPDFで用意しておくと採用されやすい。

### 海外（フェーズ2以降）
- i18n(EN)を磨いた上で Hacker News「Show HN」と Product Hunt。海外トラクションは
  博士出願で特に効く。投稿文はデモ動画＋「教育用・非営利・オープンな動機」を前面に。

## 2. フェーズ別の打ち手

### フェーズ1（〜20人）: 手渡し
- 知人・フォロワー・研究者への個別フィードバック依頼（OUTREACH.md 参照）で最初の20人。
- 全員と話せる規模を活かし、5人以上にインタビュー。改善が最良のマーケティング。
- この期間の X は「開発ログ」路線（作っている過程自体をコンテンツにする）。

### フェーズ2（〜50人）: 週次の定常運転
- 週替わりシナリオ結果スレッドを毎週定時に開始（曜日固定で習慣化）。
- 共同カウンター稼働後は「全国で N の街が守られました」を月次で報告。
- Zenn記事1・2を公開。教員1〜2人への授業利用打診（1クラスでフェーズ2目標に届く）。

### フェーズ3（〜100人）: 物語の完成
- 「シミュレーションから現実へ」（現実アクション機能）をプレスの切り口として打ち出す。
- Zenn記事3、学会LT、可能ならShow HN。
- 100人到達時に「数字の揃った物語」をまとめる: 累積で守られた街の数 / 週次リテンション /
  共有経路の内訳 / 現実アクション件数。→ そのまま就活・出願のポートフォリオ資料に。

## 3. コンテンツカレンダー（雛形）

> **投稿文の実テンプレは `docs/x-post-templates.md`**（月/木/日の雛形＋W31〜W34の月曜告知下書き＋送信前チェックリスト）。

| 曜日 | 内容 |
|---|---|
| 月 | 週替わりシナリオ開始の告知（アプリ通知と同時刻） |
| 木 | 中間経過（生存率、面白い失敗パターン） |
| 日 | 結果スレッド（集合統計＋構造の学び＋来週の予告） |
| 随時 | 開発ログ、発見図鑑の1項目解説、ユーザーの共有カードの引用RT（許可を得て） |

## 4. リスクと対処

- **炎上リスク**: 実名・実在地域を想起させる投稿をしない。進行中の政局に運営から言及しない。
  エチケットカード（「特定の誰かではなく構造の話」）を共有物すべてに残す。
- **スパム判定**: リプ営業をしない。DMは個別・少数・パーソナライズのみ（OUTREACH.md）。
- **無人の広場**: ユーザーが少ない段階で「みんなの◯◯」系機能を見せない（累積型のみ見せる）。
- **測定倫理**: Cookieレス・個人非特定の計測を維持し、その旨をアプリ内に明記（思想との一貫性）。

## 5. KPI ダッシュボード（週次で見る数字）

- 新規訪問（Web）/ インストール（アプリ、ストア統計）
- weekly_start ユニーク（=週次リテンションの実体）
- share_x / share_line / card_saved の比率（どの経路が人を連れてくるか）
- 巻き戻し実行率（体験の核が使われているか）
- 記事PV・学会接点数・引用/言及（研究者チャネルの健康度）
