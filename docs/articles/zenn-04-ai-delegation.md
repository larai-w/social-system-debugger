## この記事の主題

個人開発のシミュレーターを、**AIサブエージェントに実装を委任しながら**、人間（自分）は**仕様書・レビュー・受け入れ検証・CI**に集中する、という体制で作っています。

この記事は「AIにコードを書かせたら速かった」という話ではありません。むしろ逆で、**AIに実装を任せられるようにするために、人間側で何を統制の仕組みとして用意したか**の記録です。ツールは Claude Code のサブエージェント機能を使っています。

対象アプリは教育用シミュレーター（社会や地域インフラが**どう壊れるか**をスライダー操作で体験する非営利・無料のWebアプリ）。本体は単一の巨大な `index.html` を、バンドラなしでモジュール分割した構成で、Web（GitHub Pages / CloudFront）とネイティブ（Capacitor）で配信しています。

## 1. 前提: 何を「委任」と呼んでいるか

体制は2階層です。

- **メインセッション（人間＋上位モデル）** … 仕様を書く／成果物をレビューする／受け入れコマンドを実行する／git commit する。**判断と統制**を担う。
- **サブエージェント（Claude Code の Agent 機能で起動）** … 仕様書の範囲内で、**自己完結した実装**を書く。コミットはしない。

重要なのは、「委任」＝「丸投げ」ではないこと。委任しているのは**実装という作業**であって、**何を作るか（仕様）・良し悪しの判断（レビュー）・受け入れの合否（検証）**は一切委任していません。ここを混同すると品質が崩れます。

このプロジェクトでは、この体制を明文化した委任プロトコルをリポジトリ内（`CLAUDE.md` / `AGENTS.md`）に置き、毎セッション自動で読ませています。「その場のノリ」ではなく、**再現可能なルール**として運用しているのがポイントです。

## 2. 仕様書の型

委任の入口は仕様書です。曖昧な依頼は曖昧な成果物を生むので、テンプレート（`docs/task-spec-template.md`）を1つ決めて、必ずこの形で渡します。骨子はこうです。

```markdown
## タスク: <一行で>

### 目的（なぜやるか）
<戦略文書のどこに根拠があるか1〜2行>

### まず読むファイル（この順で）
1. CLAUDE.md（プロジェクト制約。特に integrity と i18n）
2. <対象に関係する既存ファイル・参考実装>

### 作成/変更してよいファイル（これ以外は触らない）
- <パスを列挙。新規/編集を明記>

### 要求仕様
- <箇条書きで具体的に。曖昧語を使わない>

### 禁止事項
- git commit / push（レビューは親が行う）
- 仕様外ファイルの変更・削除

### 受け入れ条件（全部満たすまで完了と報告しない）
1. `npm run check` が green（prettier/eslint 含む）
2. <タスク固有の検証コマンドと期待結果>
```

効いているのは、地味な3欄です。

- **「作成/変更してよいファイル（これ以外は触らない）」** … 触ってよい範囲を明示すると、レビューの差分が予測可能になる。想定外のファイルが変わっていたら、それ自体がレッドフラグ。
- **禁止事項** … とくに「コミットしない」。判断のゲートを人間側に残すため。
- **受け入れ条件** … 「完成」の定義を仕様書の側に固定する。感覚で「できた」と言わせない。

## 3. 統制の3原則

任せられるようにするために、人間側で守っているルールが3つあります。

### 原則① サブエージェントはコミット禁止

サブエージェントは実装するが、**git commit / push はしない**。マージの判断は必ず人間が握ります。これがあるので、成果物が期待外れでも、そのまま履歴に混ざることがありません。「レビューを通ったものだけが main に入る」というゲートを、体制として担保しています。

### 原則② 受け入れコマンドは親が必ず再実行する

これは実運用で得た、いちばん大事な学びです。

サブエージェントは「`npm run check` は green でした」と報告してきます。**それを鵜呑みにしない。** 親セッションで**必ず同じコマンドを再実行**し、自分の目で green を確認してからコミットします。

理由は次章の実話に直結します。サブエージェントの実行環境では、そもそも Bash が拒否されて**テストを一度も走らせられないケース**があるからです。自己申告の green が、実際には「一度も検証していない」を意味することがある。だから受け入れは常に親側で取り直す、というのを恒久ルールにしました。

### 原則③ 制約は「レビュー」ではなく「CI」に落とす

人間のレビューは見落とします。だから、このプロジェクトの絶対制約は、レビュー頼みにせず**機械が落とす仕組み**にしています。

- **i18n の完全性** … 新規のユーザー向け文字列は日英両方に必要。これを目視ではなく、辞書キーの対称性・全 `data-i18n` キーの存在をテスト（`tests/i18n-completeness.test.mjs`）で検証。
- **シナリオの到達可能性** … 配信するシナリオが「開始と同時にクリア済み」や「永久にクリア不能」になっていないかを、実装の計算関数を抽出して headless 実行で検証（`tests/weekly-reachability-p234.test.mjs` ほか）。
- **Console エラーゼロ** … 全タブ遷移・プリセット・スライダー操作・ショック注入・エクスポートまでを Playwright で実行し、`pageerror` / `console.error` がゼロであることを検証（`npm run verify`）。

これがあると、委任の安全性が上がります。**サブエージェントが仕様を勘違いしても、CIが構造的に落とす**からです。人間のレビューは「意図が合っているか」に集中でき、「日英片方だけ追加していないか」のような機械的な穴はCIが埋めます。

## 4. 実際に起きたこと（事実ベースのエピソード3つ）

### (a) サンドボックスで書き込みが拒否された → 「全文納品→親転記」の確立

サブエージェントの実行環境で、Bash に加えて**ファイルの Write まで拒否される**ケースに遭遇しました。エージェントは実装できないのに、タスクは自己完結型です。

ここで決めたフォールバックが、「**実装不能なら、完成した成果物の全文＋実コードに基づく検証トレースを『報告』として納品し、親がそれを転記して受け入れる**」というもの。あるスプリントでは、週替わりシナリオの設計と、実装コードから数値を追った到達可能性のトレース表までをエージェントが報告として作り、ファイルへの書き込みと受け入れコマンドは親が代行しました。

このフォールバックを恒久プロトコルに昇格させたことで、「環境がWriteをブロックしてもAIの成果を捨てずに済む」状態になりました。（その後、権限設定で Write/Edit を許可し、直接ファイル編集できたスプリントもあります。環境差はあるので、両方の運用を残しています。）

### (b) セッション上限で停止 → 成果物は作業ツリーから回収できた

別のスプリントで、サブエージェントが**セッションの使用量上限に達して、完了報告を返す前に停止**しました。普通なら「作業ロスト」ですが、そうはなりませんでした。

**エージェントが書き込み済みだったファイルは、作業ツリーに残っていた**からです。親セッションが `git status` から未コミットの成果物を回収し、レビューして受け入れコマンドを走らせることで、タスクを完遂できました。

学びは単純です。**コミット権を親に集約し、成果物はワークツリーに残す設計にしておくと、エージェントが途中で死んでも回収可能**になる。これは原則①（サブはコミットしない）の副次的な恩恵でもあります。

### (c) 委任コンテンツの品質を守った決定打 ＝ 到達可能性CIが導入即バグを検出

いちばん効いた話です。

週替わりシナリオは、AIに委任して量産しているコンテンツの1つです。各シナリオには「このゴール条件を満たせばクリア」という達成条件があり、**開始状態でクリア可能かどうか**は、実装の計算式を人間が手でトレースして検証していました。手計算は退屈で、間違えます。

そこで、この検証を CI に置き換えました（`tests/weekly-reachability-p234.test.mjs`）。ui.js の実装関数をソースから抽出して headless 実行し、(a) 開始パラメータでクリアにならないこと、(b) 探索空間内でゴールに到達可能なこと、を機械が検証します。

**このテストは、導入した直後に実バグを1件検出しました。** 委任で作られたあるシナリオ（PAGE 3 系）が、開始時点で判定メトリクスが達成側に振れていて、**開始と同時にクリアになる**設計になっていたのです。表示上は普通に見えるので、目視レビューでは気づけないタイプでした。テストが指摘したので、探索深さのガードを追加して修正できました。

これで、委任シナリオの「数値トレースを親が手検証する」工程そのものが不要になりました。**AIの生産量が増えても、品質はCIが構造的に守る**——この形が、委任を安心して回すための決定打でした。

## 5. 数字で見る運用

事実として記録できている範囲の数字です（未計測のもの、たとえば利用者数などはここには書きません）。

- **委任した実装タスク: 19件。** 仕様書を書いて Claude Code のサブエージェントに渡し、レビューと受け入れを親が担う、という体制で完了したものの数です。
- **並行実行。** 1スプリントで、独立した委任タスクを複数、同時に走らせることもありました（例: サブエージェント3件＋人間直営2件の並行）。依存のないタスクを並列化できるのが、自己完結タスクを委任する利点です。
- **CIで守っている制約のテスト数: 27件。** i18n完全性・到達可能性・共有URL・エンジンのユニットテストなどを含みます。委任が増えても、この一式が毎回 green であることを受け入れ条件にしています。
- **受け入れは100%親が再実行。** 自己申告の green は、実行環境の都合で「未検証」を意味することがあるため。

数を強調したいのではなく、**「委任した数」より「委任してもCIが守る仕組みの数」を増やしてきた**、という比率が要点です。

## 6. 向く仕事 / 向かない仕事

すべてを委任しているわけではありません。基準を明確に分けています。

**委任してよい（自己完結タスク）**

- 新規の独立ファイル（`docs/` の記事、`scripts/` の生成・検証スクリプト、`content/` のデータ、単体の静的HTMLページ など）
- 翻訳・データ作成
- 既存の型（仕様書テンプレ・命名規約）に沿って**追加**する作業

**委任しない（人間が直営する）**

- 本体コアの外科手術（巨大な `ui.js` の内部改修など、既存の暗黙依存が絡むもの）
- i18n辞書や Service Worker のキャッシュ版数など、**複数箇所の整合**が絡む改修
- プロジェクトの文脈そのもの（進捗ログ・次のタスク・統制ルールの更新）

境界は「**成果物が仕様書の範囲で閉じるか**」です。閉じるなら委任、既存コードの網の目に手を入れるなら直営。前者はレビューの差分が予測可能で、CIで守りやすい。後者は文脈の切断が退行を生むので、人間が握ります。

## 7. まとめ

AIに実装を委任してみて、いちばん実感が変わったのはここです。

**速くなったのは「書く」工程だが、価値が出たのは「書かない」工程を仕組み化したことだった。**

- 何を作るかを、**曖昧語なしの仕様書**にできるか
- 「完成」の定義を、**受け入れ条件**として先に固定できるか
- 守りたい制約を、**レビュー頼みにせずCIに落とせる**か
- AIの自己申告を、**人間側で必ず取り直す**規律があるか

コードを書く力そのものより、**仕様化・検証・統制の力**が効く——というのが、この体制を回してみての結論です。逆に言えば、その土台がないままAIに実装を任せると、品質を担保する場所がどこにも無くなります。「AIがいればエンジニアは要らない」という話ではなく、**エンジニアの仕事の重心が「実装」から「仕様と検証の設計」に移る**、という実感に近いです。
