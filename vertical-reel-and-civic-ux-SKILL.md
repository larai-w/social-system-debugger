# 縦型リール制作 & 市民向けアプリUX設計スキル

社会システム・シミュレーター「社会＆地域インフラ・デバッガー」の開発で確立した、
再利用可能なノウハウ集。①スマホ縦型リール（HTML製プロモ動画）の作り方、
②市民向けアプリのUX原則、③Claude Codeへの実装指示の書き方、の3部構成。

---

## PART 1: 縦型リール制作（スマホサイズHTML動画）

### 基本アーキテクチャ

単一HTMLファイル + canvas。ユーザーがスマホで全画面表示→画面録画すると
そのまま9:16のSNS動画になる。動画生成AIより速く、修正が容易で、実アプリと
同じデザイントークンを使えるのが利点。

```
キャンバス: CW=420, CH=740（9:16近似）
スケーリング: devicePixelRatio対応のfit()関数、コンテナ幅にCSS縮尺
時間管理: マスタークロック t（秒）、DUR（総尺）、フェーズ境界定数
  例: const T_CARD=3.5, T_SHOCK=14.5, T_CONC=38;
イージング: ss(a,b,x) = smoothstep（全アニメの補間に使う）
ループ: requestAnimationFrame + dt上限0.05秒（タブ復帰時の暴走防止）
必須UI: PAUSE/RESTARTボタン、プログレスバー（下端4px・グラデ）、
  REC●点滅（録画感の演出）、終了時「↺ RESTARTで最初から」
```

### 尺と構成の使い分け

- **30秒（共有用）**: フック2秒 → 実験1つだけ → 結論 → エンドカード。
  転送される動画は「無音で完結・実験は1つ・全キャプション1〜4秒で読める」が鉄則
- **50〜70秒（単体ページ紹介）**: メリハリ4拍構成（下記）
- **110秒（フルイントロ）**: 全機能横断。自分で見る用。SNS転送には長すぎる

### メリハリの4拍構成（静→転→動→断→復）

1. **静**: ゆっくりしたログ（1.6秒間隔）、穏やかなノードの呼吸、緑基調
2. **転**: 全画面フラッシュ1発（0.15秒）＋警告テキスト（例: ⚡ BLACK SWAN INBOUND）
3. **動**: 画面シェイク（`cx.translate(rand*shake)`）、マシンガンログ（0.2〜0.5秒間隔）、
   赤ビネット（radialGradientを外周に）、スタンプ演出
4. **断**: 白フラッシュ→暗転→一言だけ（「— 奪還プロトコル —」）の1秒の溜め
5. **復**: 色が緑に反転、ログ減速、SURVIVEDスタンプ

ログの流速・ノード呼吸速度・明滅をすべてフェーズに同期させると、
無音でも緩急が伝わる。

### 定番パーツの実装パターン

- **ゴーストカーソル**: キーフレーム配列 `{t,x,y,click}` を smoothstep補間。
  白丸＋シアングロー、クリック時に拡大リング（clickPulse減衰）。
  「操作できるアプリだ」と一目で伝える最重要パーツ
- **スタンプ**: `save→translate→rotate(-0.08)→scale(拍動)` で
  枠付き大文字（CRASHED/SURVIVED）を叩きつける
- **キャプション**: 中央下部、黒地丸角ボックス＋measureTextで幅自動
- **判定バナー/チャプターカード**: フェードイン→1.5秒保持→フェードアウト
- **エンドカード**: 結論2行（対句が強い: 「効率は平時の速さ。冗長性は最悪の日の命。」）
  → アプリ名枠 → CTA点滅 → URL差し替え枠 `[ アプリURLをここに ]`

### 忠実性ルール（プロモ動画の生命線）

- **画面に映るUI文言＝実機の文言**。ラベル・ボタン名・ログ・プリセット値・
  判定式は、必ず実アプリのコードをgrepして実物を転記する。意訳・創作は
  「動画と実物が違う」というユーザーの信頼毀損に直結する
- 数値アニメーションは実エンジンの再実行ではなく「実装の因果関係に沿った手付け」
  でよいが、その場合SNS投稿文に「※演出用に簡略化したデモ映像です」を添える
- 完全一致が必要なら、アプリ本体に「デモモード（自動再生）」を実装して
  本物のエンジンで録画するのが最強
- リールのコードをアプリに移植してはいけない（フェイク値のハードコード）。
  移植するのは「演出手法」だけで、対象は本物のDOM・本物の関数にする
  （＝初回ガイドツアー、共有リンクプレビューとして実装）

### 政治的センシティビティの二本立て戦略

進行中の実際の問題と共鳴する題材は、拡散力とリスクが比例する。
- **安全版**（歴史題材・誰の現在も刺さない）を先に出す
- **共鳴版**（時事と重なる題材）は免責を強化して後発
- 全編固定フッター「※特定の誰かではなく、どの社会にも起こる構造の話です
  （架空シミュレーション）」を必ず入れる。実在の人名・県名は使わず
  「住民」等の抽象語に統一

## PART 2: UX設計原則（市民向け・社会派アプリ）

### 二人称化: 「物語の言葉」と「計器の言葉」を区別する

- 共有メッセージが二人称・感情的（「思い当たって怖い」）なのに、
  アプリ内が三人称・無機質（SECTOR-T COLLAPSE）だと、リンクを踏んだ人が
  温度差で冷める。**送る言葉と触る言葉のトーンを揃える**
- ただし全部を擬人化しない。書き換えるのは**物語の言葉**（ヘッダー副題、
  判定バナー大見出し、導入文、結果カードのタイトル）だけ。
  **計器の言葉**（ゲージ名・スライダー名・ターミナルログ・プリセット名）は
  正確なデータ読解のために不変とする
- 判定バナーは二層構造にする: 大見出し「あなたの街は、崩壊しました」＋
  小さく技術ラベル「SECTOR-T COLLAPSE」。感情と構造の両方を1画面で

### 3状態バナー: 危機／中立／模範

- 「悪い時だけ警告が出る」設計は片手落ち。模範状態には同格の視覚強度で
  緑のポジティブバナーを出す（地味だと"おまけ"に見える）
- ただし中間状態では何も出さない（常時表示はありがたみを殺す）
- ポジティブにも2トーンある: 平常運転の「今のところ平穏です」（淡々）と、
  試練を乗り越えた「生き残った。」（達成）。混ぜると参加賞っぽくなる
- 全ページで初期状態のバナー有無を揃える（1ページだけ祝福される非対称は
  バグに見える）

### 情報密度: 2階層アコーディオン

- 1ページに入力要素14個は初見に多すぎる。「主要操作（常に表示）＋
  詳細設定（折りたたみ）」に分ける。プリセットボタンは最も手軽な入口なので
  絶対に畳まない
- DOMは残してdisplay:noneで隠すだけにし、計算ロジックに影響させない

### 研究者と一般層の両立: モードで分ける、妥協しない

- デフォルト画面を1つに固定して綱引きするのではなく、
  「🎓 研究者向け表示」トグル（全アコーディオン一括展開＋localStorage永続化）
  で両方に「自分向けだ」と感じさせる
- 知的に見える要素は「専門用語の深さ」であって「同時に見えるスライダー本数」
  ではない。折りたたみはむしろ設計の丁寧さに見える

### ゲーミフィケーション: 収集型のみ、搾取型は禁止

- アプリ自体がドーパミンループを批判するなら、ストリーク・時間制限・
  通知プッシュ・「まだ◯個未達成」の不安煽りは自己矛盾。実装しない
- 採用するのは**発見ログ**: 注目すべき状態（クラッシュ、デッドロック、生存…）
  に「バッジ名＋1行の学び」をセットで定義。
  例: 「🏚 他責デッドロックを目撃した — 直せるのに、誰も直さない状態を見た。」
- 未発見は「❓ ???」で隠す（好奇心は刺激、誘導はしない）。
  カウンターは中立表記「📖 発見ログ (x/12)」
- 滞在時間は「触るたびに新しい発見と短い学びがある」ループだけで伸ばす

### 共有設計: 「意見」ではなく「実験」を運ぶ

- 分断環境で意見を送ると受け手が陣営判定モードに入る。送るのは
  「このパラメータでこうなった、面白くない？」という**再現可能な実験**
  （＝パラメータ入りURL）
- 共有ボタンは常設1個より「感情のピーク直後」（判定確定の瞬間）に出す
  文脈ボタンの方が数倍押される
- テンプレ文面は「フック1行＋説明1行＋URL、URL手前120字以内」。
  最強の汎用フックは「誰の話もしてないのに、いろいろ思い当たるのが怖い」
  （抽象性こそ送りやすさの源泉、と送り手に再確認させる）
- 全テンプレ末尾に固定免責1行。実在の人名・陣営を示す語は禁止
- **共有作法カード（3か条）**: ①人名を、添えない ②答えを、言わない
  ③好奇心だけ、添える。＋「※推奨の一例です。必ず守るルールではありません」。
  結果カードPNGの2枚目として毎回自動添付し、「送る瞬間に作法が目に入る」
  設計にする（人はガイドを読みに行かない）
- **URLは常に1本**。チャネル別リンクを作るとパラメータ復元が壊れる。
  プレビューの出し分けはOGPメタタグに任せる
- ページ単体へのクリーンな直接リンク（?tab=Nのみ）と、状態フルの
  シナリオ共有URLは別ボタンとして明確に分ける。タブ切替時は
  history.replaceStateでアドレスバーを同期

### フィードバック: 窓口を2つに分ける

- 一般向け: モーダル内フォームで送信完結（Formspree等の
  サーバーレスフォーム）。mailto:だけに頼らない（スマホで離脱される）
- 専門家向け: GitHub Issues（公開・記録蓄積・技術者は慣れている）。
  研究者モードON時はこのリンクを格上げ表示する

## PART 3: Claude Codeへの実装指示プロンプトの書き方

大規模化した単一HTMLアプリを壊さず差分実装させるための定型:

1. **「既存実装済み・作り直し禁止」リストを冒頭に明記**
   （関数名を列挙。これが無いと丁寧な既存実装を上書きされる）
2. **「まずindex.htmlを読み、実際の関数名・変数名・data-i18nキーを
   特定し、実装計画を短く提示してから着手」で必ず締める**
   （推測での命名を防ぐ。効果絶大）
3. スコープは外科的に。「今回は見た目のみ、文言・ロジックには触れない」
   等の否定形の制約を明記
4. 文言の完全一致が必要な箇所は「新規に書き起こさず、コード内の
   実文字列（sg_dont等）をコピー元とする」と指定し、実装後に
   自動照合させる
5. 外部サービス連携はプレースホルダー定数
   （`const FEEDBACK_ENDPOINT='...YOUR_FORM_ID'`）で完成させ、
   ID差し替えだけで有効化できる形に
6. 受け入れ条件を番号付きテストとして列挙（①〜⑥）。
   「既存機能が変わっていないこと」を必ず1項目入れる
7. Claude Codeの環境ではcanvas描画とWeb Share APIの実挙動を検証できない。
   「プレビュー用HTMLを作って描画結果を見せて」と依頼し、
   最終確認は実機（スマホ）チェックリストとして人間側タスクに切り出す。
   恒久検証にはPlaywright入りDockerテストハーネス
   （Dockerfile.test / navigator.shareモック / PNG出力）を別途用意する
8. 参考デザインは文章で説明せずコードごと渡す。その際
   「参照するのはdrawEtiquette()とrrect()のみ、他のUIは対象外」と
   範囲を絞る
